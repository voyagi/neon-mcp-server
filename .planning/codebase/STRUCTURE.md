## Directory Purposes

**src/**
- Purpose: All source TypeScript code, organized by layer
- Contains: Entry point, server setup, data access tools, type definitions, external integrations
- Key files: `index.ts` (startup), `server.ts` (registration), `lib/` (integration), `tools/` (implementations)

**src/lib/**
- Purpose: Shared utilities and cross-cutting concerns
- Contains: Supabase client initialization, domain type definitions
- Key files: `supabase.ts` (DB client), `types.ts` (TypeScript interfaces)

**src/tools/**
- Purpose: MCP tool implementations for CRUD and analytics operations
- Contains: (Empty, planned) Tool handler functions for customers, tickets, products, analytics
- Key files: (Not yet created)

**src/resources/**
- Purpose: MCP resource implementations for read-only data exposure
- Contains: (Empty, planned) Schema reflection and discovery resources
- Key files: (Not yet created)

**dist/**
- Purpose: Compiled JavaScript output from TypeScript compilation
- Contains: .js, .d.ts, and .map files generated by `tsc`
- Generated: Yes
- Committed: No (in .gitignore)

**seed/**
- Purpose: Database initialization and setup documentation
- Contains: Demo data as SQL, setup instructions
- Key files: `seed.sql` (10 customers, 10 products, 10 tickets for demo)

**.planning/codebase/**
- Purpose: GSD analysis documents (architecture, structure, conventions, testing, concerns)
- Contains: ARCHITECTURE.md, STRUCTURE.md, etc.
- Generated: Yes (by GSD mapper)
- Committed: Yes

## Key File Locations

**Entry Points:**
- `src/index.ts`: Node.js process entry point; initializes server and stdio transport. Called by `node dist/index.js` from Claude Desktop config.

**Configuration:**
- `tsconfig.json`: TypeScript compiler options (ES2022, Node16, strict mode)
- `biome.json`: Linter and formatter rules (Biome replaces ESLint + Prettier)
- `.env.example`: Template showing required env vars (SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY)

**Core Logic:**
- `src/server.ts`: Server setup, tool and resource registration (currently a stub with TODO comments)
- `src/lib/supabase.ts`: Singleton Supabase client, reads credentials from env
- `src/lib/types.ts`: TypeScript interfaces for Customer, Ticket, Product entities

**Testing:**
- No test files yet; `package.json` has no test script

**Data & Seeds:**
- `seed/seed.sql`: Demo data for the CRM—10 customers (active/inactive/lead), 10 products (support/service/product), 10 tickets (various statuses/priorities)

## Naming Conventions

**Files:**
- Source files: `camelCase.ts` (e.g., `supabase.ts`, `types.ts`)
- Entry point: `index.ts`
- Server setup: `server.ts`
- Directories: `kebab-case` or `lowercase` (e.g., `src/lib/`, `src/tools/`, `src/resources/`)

**Functions:**
- Tool handlers: `camelCase` (e.g., `listCustomers`, `getTicket`, `createCustomer`)
- Utility functions: `camelCase` (e.g., `initializeSupabase`)
- Main functions: `main()` for entry point

**Variables & Constants:**
- Const: `camelCase` or `UPPER_SNAKE_CASE` for env-derived constants
- Example: `const supabaseUrl = process.env.SUPABASE_URL`

**Types:**
- Interfaces: `PascalCase` (e.g., `Customer`, `Ticket`, `Product`)
- Enums: `PascalCase` (not used yet, but would be e.g., `TicketStatus`)
- Type aliases: `PascalCase` (not used yet)

## Where to Add New Code

**New MCP Tool:**
1. Create new file in `src/tools/` (e.g., `src/tools/customers.ts`)
2. Export async function matching tool signature: `export async function listCustomers(params: InputSchema): Promise<ToolResult>`
3. Use `src/lib/supabase.ts` to query database
4. Reference `src/lib/types.ts` for return type definitions
5. Register tool in `src/server.ts` with schema and handler reference

**New MCP Resource:**
1. Create new file in `src/resources/` (e.g., `src/resources/schema.ts`)
2. Export function that reads database schema and formats as MCP resource
3. Register resource in `src/server.ts` with URI (e.g., `schema://tables`) and content handler
4. Return structured JSON describing tables and columns

**New Shared Type:**
1. Add TypeScript interface to `src/lib/types.ts`
2. Example: If tickets need a new field, add to `Ticket` interface
3. Update any tools that read/write that entity

**Utilities:**
1. If utility is cross-cutting (used by multiple tools), add to `src/lib/`
2. Create new file (e.g., `src/lib/validation.ts`) or extend existing
3. Export named functions; import in tools as needed

**Dependencies:**
1. Update `package.json` manually or use `npm install`
2. Zod is already available for validation (not yet used)
3. MCP SDK and Supabase client already installed
